## Project Overview

**What:** A single-page personal portfolio website for Brett Hardiman — a technologist specializing in agentic AI workflows, formerly at Booz Allen Hamilton.

**Who it is for:** Recruiters at tech companies and technical peers evaluating Brett's background.

**Primary goal:** Drive visitors to connect with Brett on LinkedIn.

**Deployment target:** GitHub Pages at `brett-hardiman.github.io/personal-portfolio` (no custom domain for now).

**Tech stack:** Vanilla HTML5, CSS3, and JavaScript (ES6+). No frameworks, no build tools, no bundlers. Every file is served as-is by GitHub Pages.

**Design language:** Spotify's dark aesthetic — dark backgrounds (#191414, #000000), green accents (#1DB954), light text (#FFFFFF, #B3B3B3). Single-page scrolling layout inspired by [briceclain.com](https://briceclain.com/en/).

## Architecture Overview

### Single-Page Architecture

The entire site is a single `index.html` file with clearly demarcated `<section>` elements. Navigation is handled via smooth-scroll anchor links. There are no route changes, no page loads after initial render.

```
index.html
  ├── <header>        — Fixed navigation bar + scroll progress indicator
  ├── <section#hero>   — Name, title, tagline, CTA
  ├── <section#about>  — Professional background
  ├── <section#skills> — Technology badges/icons
  ├── <section#experience> — Work timeline
  ├── <section#projects>   — Project showcase cards
  ├── <section#testimonials> — Colleague quotes (nice-to-have)
  ├── <section#education>    — Academic background (nice-to-have)
  ├── <section#contact>      — LinkedIn CTA + social links
  └── <footer>         — Copyright, social icons
```

### CSS Architecture

CSS is split into modular files for maintainability. No preprocessor — plain CSS with custom properties (CSS variables) for the design tokens.

**Loading strategy:** A single `styles.css` file that `@import`s the modular files. For GitHub Pages performance, this is acceptable given the small number of files. If performance profiling later shows render-blocking issues, the files can be concatenated manually.

```css
/* styles.css — entry point */
@import 'css/variables.css';
@import 'css/reset.css';
@import 'css/typography.css';
@import 'css/layout.css';
@import 'css/nav.css';
@import 'css/hero.css';
@import 'css/about.css';
@import 'css/skills.css';
@import 'css/experience.css';
@import 'css/projects.css';
@import 'css/testimonials.css';
@import 'css/education.css';
@import 'css/contact.css';
@import 'css/footer.css';
@import 'css/animations.css';
@import 'css/responsive.css';
```

### JavaScript Architecture

JavaScript is split into focused modules loaded with `type="module"` for clean scoping. No bundler required — modern browsers support ES modules natively, and GitHub Pages serves them correctly.

```
js/
├── main.js            — Entry point, initializes all modules
├── nav.js             — Scroll-based nav highlighting, mobile menu toggle
├── scroll-progress.js — Thin progress bar at top of viewport
├── scroll-reveal.js   — Intersection Observer for fade-in animations
├── parallax.js        — Mouse-tracking parallax effect in hero
└── projects.js        — Project card hover effects
```

**Graceful degradation:** All content is visible and readable with JavaScript disabled. JS only adds animations, hover effects, and interactive enhancements. The nav links work as standard anchor links without JS.

#### 1.2 — CSS Design System

- `variables.css` — All design tokens from the Design System section above
- `reset.css` — Modern CSS reset (box-sizing, margin/padding normalization, smooth scrolling, image defaults)
- `typography.css` — Font-face declarations, heading styles (h1-h6), body text, link styles, monospace styling
- `layout.css` — Container class (max-width: 1200px, centered), section padding, CSS Grid and Flexbox utility patterns

#### 1.3 — Fixed Navigation Bar

- Fixed position at top of viewport
- Semi-transparent dark background with backdrop blur
- Logo/name on the left
- Section links on the right (horizontal list)
- Active link highlighting based on current scroll position (JS added in Phase 3, but the CSS classes are defined here)
- Mobile hamburger menu: hidden on desktop, replaces nav links on mobile
- Hamburger toggle is CSS-only for basic show/hide (JS enhancement in Phase 3)

```css

  position: fixed;
  top: 0;
  left: 0;
  right: 0;
  z-index: 100;
  background-color: rgba(25, 20, 20, 0.9);
  backdrop-filter: blur(10px);
  -webkit-backdrop-filter: blur(10px);
  border-bottom: 1px solid var(--color-border);
}
```

#### 1.4 — Responsive Grid Foundation

- Mobile-first approach: base styles target mobile, media queries add complexity
- Container widths: full-width with padding on mobile, max-width on larger screens
- Section layout: single column on mobile, expanding to multi-column where appropriate on larger screens
- CSS Grid for the project cards layout and skills grid
- Flexbox for navigation and inline element alignment

### Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| CSS loading | `@import` chain from `styles.css` | Simpler than managing multiple `<link>` tags. Acceptable performance for a small site with ~15 CSS files. |
| JS loading | Single `<script type="module">` | ES modules provide native scoping. No bundler needed. Deferred by default. |
| CSS methodology | BEM-lite (block-element pattern, no strict modifiers) | Keeps class names readable without over-engineering for a single-page site. |
| Font loading | Google Fonts `<link>` with `display=swap` | Fast, reliable CDN. `display=swap` prevents FOIT (flash of invisible text). |

### Dependencies
- None — this is the first phase.

### Risks and Mitigation

| Risk | Impact | Mitigation |
|------|--------|------------|
| CSS `@import` causes render-blocking chain | Slow initial paint on very slow connections | Acceptable for portfolio-scale site. If profiling shows issues, concatenate files manually before deploy. |
| `backdrop-filter` not supported in older browsers | Nav background fully transparent | Fallback: solid background color without blur via `@supports` query. |

### Definition of Done
- `index.html` loads in Chrome, Firefox, and Safari without errors
- All CSS variables render correctly (colors, fonts, spacing visible)
- Navigation bar is fixed at top, visually styled, and links scroll to correct sections
- Layout is responsive: single column on mobile (< 768px), multi-column on desktop
- Mobile hamburger menu is visible on small screens (toggle can be non-functional until Phase 3)
- HTML passes W3C validation with no errors
- All text is legible against dark backgrounds (contrast ratio >= 4.5:1 for body text, >= 3:1 for large text — WCAG AA)

## Phase 3 — Animations and Interactivity

### Objective

All enhancements are progressive — the site remains fully functional without JS.

### What Gets Built

#### 3.1 — Scroll Progress Indicator (`scroll-progress.js`)

A thin (3px) green bar at the very top of the viewport that fills from left to right as the user scrolls down the page.

```javascript
// scroll-progress.js
export function initScrollProgress() {
  const progressBar = document.querySelector('.scroll-progress');
  if (!progressBar) return;

  function updateProgress() {
    const scrollTop = window.scrollY;
    const docHeight = document.documentElement.scrollHeight - window.innerHeight;
    const progress = docHeight > 0 ? (scrollTop / docHeight) * 100 : 0;
    progressBar.style.width = `${progress}%`;
  }

  window.addEventListener('scroll', updateProgress, { passive: true });
  updateProgress();
}
```

#### 3.2 — Scroll-Triggered Reveal Animations (`scroll-reveal.js`)

Uses the Intersection Observer API to add a `.revealed` class to elements as they scroll into view. Elements start with `opacity: 0` and `transform: translateY(20px)`, then animate to their final position.

**Implementation approach:**
- Any element with `data-reveal` attribute gets observed
- Optional `data-reveal-delay` attribute for stagger effects (e.g., skill badges appearing one after another)
- Threshold: 0.15 (element is 15% visible before triggering)
- Once revealed, the observer disconnects from that element (one-time animation)

```javascript
// scroll-reveal.js
export function initScrollReveal() {
  const elements = document.querySelectorAll('[data-reveal]');
  if (!elements.length) return;

  const observer = new IntersectionObserver((entries) => {
    entries.forEach(entry => {
      if (entry.isIntersecting) {
        const delay = entry.target.dataset.revealDelay || 0;
        setTimeout(() => {
          entry.target.classList.add('revealed');
        }, Number(delay));
        observer.unobserve(entry.target);
      }
    });
  }, { threshold: 0.15 });

  elements.forEach(el => observer.observe(el));
}
```

**CSS:**
```css
[data-reveal] {
  opacity: 0;
  transform: translateY(20px);
  transition: opacity 0.6s ease-out, transform 0.6s ease-out;
}

[data-reveal].revealed {
  opacity: 1;
  transform: translateY(0);
}
```

**Accessibility consideration:** Users who prefer reduced motion should not see these animations.

```css
@media (prefers-reduced-motion: reduce) {
  [data-reveal] {
    opacity: 1;
    transform: none;
    transition: none;
  }
}
```

#### 3.3 — Hero Parallax Mouse Tracking (`parallax.js`)

Subtle parallax effect where background elements in the hero section shift slightly based on mouse position. This mirrors the briceclain.com hero.

**Implementation:**
- Track `mousemove` events on the hero section
- Apply CSS `transform: translate()` to decorative elements based on cursor position relative to viewport center
- Movement is subtle: max 15-20px shift in any direction
- On mobile/touch devices, this effect is disabled entirely (no mousemove on touch)
- Uses `requestAnimationFrame` for smooth performance

```javascript
// parallax.js
export function initParallax() {
  const hero = document.querySelector('.section--hero');
  if (!hero || window.matchMedia('(hover: none)').matches) return;

  const layers = hero.querySelectorAll('[data-parallax-speed]');
  if (!layers.length) return;

  let rafId = null;
  let mouseX = 0;
  let mouseY = 0;

  hero.addEventListener('mousemove', (e) => {
    mouseX = (e.clientX / window.innerWidth - 0.5) * 2;
    mouseY = (e.clientY / window.innerHeight - 0.5) * 2;

    if (!rafId) {
      rafId = requestAnimationFrame(() => {
        layers.forEach(layer => {
          const speed = parseFloat(layer.dataset.parallaxSpeed) || 1;
          const x = mouseX * 15 * speed;
          const y = mouseY * 15 * speed;
          layer.style.transform = `translate(${x}px, ${y}px)`;
        });
        rafId = null;
      });
    }
  });
}
```

**Decorative elements for parallax:** CSS-generated geometric shapes (circles, grid dots) positioned absolutely in the hero section. These are purely decorative (`aria-hidden="true"`).

#### 3.5 — Project Card Hover Effects (`projects.js`)

On desktop, project cards respond to mouse movement with a subtle 3D tilt effect (CSS `perspective` + `rotateX`/`rotateY`). On hover, the card also gets a green glow/border effect.

**Implementation:**
- Track `mousemove` on each card
- Calculate cursor position relative to card center
- Apply small `rotateX` and `rotateY` transforms (max 5 degrees)
- Reset on `mouseleave`
- Disabled on touch devices

```javascript
// projects.js
export function initProjectCards() {
  const cards = document.querySelectorAll('.project-card');
  if (window.matchMedia('(hover: none)').matches) return;

  cards.forEach(card => {
    card.addEventListener('mousemove', (e) => {
      const rect = card.getBoundingClientRect();
      const x = (e.clientX - rect.left) / rect.width - 0.5;
      const y = (e.clientY - rect.top) / rect.height - 0.5;
      card.style.transform = `perspective(800px) rotateY(${x * 5}deg) rotateX(${-y * 5}deg) scale(1.02)`;
    });
    card.addEventListener('mouseleave', () => {
      card.style.transform = 'perspective(800px) rotateY(0deg) rotateX(0deg) scale(1)';
    });
  });
}
```

### Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Intersection Observer over scroll listeners | Intersection Observer for reveal and nav tracking | More performant than scroll event listeners. Does not trigger layout recalculation. |
| `requestAnimationFrame` for parallax | Yes | Prevents janky rendering by syncing transforms with the browser's paint cycle. |
| Touch device detection | `window.matchMedia('(hover: none)')` | More reliable than user-agent sniffing. Correctly handles touch laptops. |
| `{ passive: true }` on scroll listeners | Yes | Tells the browser the handler will not call `preventDefault()`, enabling scroll performance optimizations. |
| `prefers-reduced-motion` respect | All animations gated | WCAG AA requirement and general best practice. Some users experience motion sickness. |

### Dependencies
- Phase 1 (HTML structure with correct IDs and classes)
- Phase 2 (content sections with `data-reveal` attributes — these attributes will be added to the HTML during Phase 2 or as part of Phase 3 implementation)

### Risks and Mitigation

| Risk | Impact | Mitigation |
|------|--------|------------|
| Parallax effect feels gimmicky or distracting | Reduces professionalism | Keep movement very subtle (max 15px). Use only on decorative elements, never on text. User testing recommended. |
| Intersection Observer not supported in very old browsers | Reveal animations never trigger, content stays invisible | The `[data-reveal]` CSS sets `opacity: 0` — if Observer fails, content is hidden. Mitigation: add a `<noscript>` style block that sets `opacity: 1` on all `[data-reveal]` elements. Also add a JS fallback that reveals all elements if IntersectionObserver is undefined. |
| Too many simultaneous animations cause jank on low-end devices | Poor user experience | Limit concurrent animations. Use `will-change` sparingly. Test on throttled CPU in DevTools. |
